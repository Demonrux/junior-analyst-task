# Задание 3: архитектура push-уведомлений

## Краткое описание

В мобильном приложении интернет-магазина «Петрушка Зеленая» требуется функционал отправки push-уведомлений разных типов (просроченная корзина, отмена заказа, рекламные рассылки и т.д.). Бэкенд построен на микросервисной архитектуре. Ниже описана верхнеуровневая схема работы.

## Компоненты системы

1. **Бизнес-микросервисы** (единый блок на схеме)  
   - Сервис заказов, сервис корзины, рекламный/маркетинговый сервис и др.  
   - Генерируют события, требующие отправки пуша.  
   - Публикуют эти события в брокер, не заботясь о том, кто их обработает.

2. **Брокер событий (Kafka / RabbitMQ)**  
   - Асинхронная очередь сообщений.  
   - Обеспечивает надёжную доставку событий от микросервисов к сервису уведомлений.  
   - Позволяет легко добавлять новых подписчиков (аналитика, логирование) без изменения кода источников.

3. **Сервис уведомлений (Notification Service)**  
   - Центральный компонент: подписывается на события брокера.  
   - Хранит push-токены устройств и настройки пользователей.  
   - Формирует payload для FCM/APNS.  
   - Отправляет уведомления через шлюзы.  
   - Управляет отложенными пушами (через Redis).  

4. **Базы данных**  
   - **PostgreSQL** – постоянное хранение: токены устройств, пользовательские настройки.  
   - **Redis** – хранилище для отложенных задач (например, «отправить напоминание о корзине через 2 часа»).

5. **Платформенные шлюзы**  
   - **FCM** (Firebase Cloud Messaging) – для Android.  
   - **APNS** (Apple Push Notification Service) – для iOS.  
   - Сервис уведомлений вызывает их API для доставки пуша на конкретное устройство.

6. **Мобильное приложение**  
   - При запуске запрашивает разрешение на push.  
   - Получает от ОС уникальный токен и отправляет его на бэкенд.  
   - Принимает входящие уведомления, обрабатывает нажатия.
